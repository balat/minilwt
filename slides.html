A talk in code

minilwt

Three tiny promise schedulers, one idea at a time.

Classic monadic Lwt  ·  effects that break the semantics  ·  effects that keep them.

Vincent Balat

Motivation

Why compare?

  • Lwt is promises + a monadic bind. The type 'a Lwt.t marks, in the signature, that a function is asynchronous.
  • OCaml 5 brought effects (Eio, Miou, ...) and fast direct-style schedulers. Can we get that speed and keep Lwt's monadic type and full compatibility?
  • Shrink Lwt to its essence and compare three cores that differ in exactly one function: bind.
Shared core — the promise

A promise, minimally

type 'a t = { mutable state : 'a state }
and 'a state = Done of 'a | Waiting of ('a -> unit) list

let return v = { state = Done v }

(* resolve a pending promise, fire its callbacks *)
let wakeup p v = match p.state with
  | Done _ -> invalid_arg "wakeup: already resolved"
  | Waiting ws -> p.state <- Done v; List.iter (fun f -> f v) (List.rev ws)

(* run a callback now if resolved, else register it *)
let on_resolve p f = match p.state with
  | Done v -> f v
  | Waiting ws -> p.state <- Waiting (f :: ws)

A promise is a mutable cell: a value, or a list of waiting callbacks.

Shared core — sleep & yield

Two ways to suspend

let ready  = Queue.create ()   (* tasks from yield *)
let timers = ref []            (* (deadline, wake) for sleep *)

let sleep delay =              (* resolves after [delay] seconds *)
  let p = { state = Waiting [] } in
  timers := (Unix.gettimeofday () +. delay, fun () -> wakeup p ()) :: !timers;
  p

let yield () =                 (* resolves on the next tick *)
  let p = { state = Waiting [] } in
  Queue.add (fun () -> wakeup p ()) ready;
  p

Each returns a pending promise and schedules its own wake-up.

Shared core — the run loop

Where it all turns: run

let rec run p = match p.state with
  | Done v -> v                          (* main promise resolved: return it *)
  | Waiting _ ->
      (if not (Queue.is_empty ready) then Queue.pop ready ()   (* a ready task *)
       else match List.sort (fun (a,_) (b,_) -> compare a b) !timers with
         | [] -> failwith "run: nothing left to run"
         | (t, wake) :: rest ->          (* else the earliest timer *)
             timers := rest;
             let dt = t -. Unix.gettimeofday () in
             if dt > 0. then Unix.sleepf dt;
             wake ());
      run p

Ready tasks first; otherwise sleep until the nearest timer. The very same loop drives all three cores.

Version 1 — classic monadic

Callbacks, no effects minilwt.ml

let bind p f =
  let r = { state = Waiting [] } in
  on_resolve p (fun v -> on_resolve (f v) (wakeup r));
  r

let ( let* ) = bind
  • bind posts a callback and returns a result promise at once — the caller keeps going, so sibling branches all start: Lwt's implicit concurrency.
  • No effects. run : 'a t -> 'a takes the promise directly.
Version 1 — bind, explained

Why two on_resolve?

let bind p f =                 (* f : 'a -> 'b t  — returns a PROMISE *)
  let r = { state = Waiting [] } in
  on_resolve p (fun v ->               (* 1. wait for p : 'a *)
    on_resolve (f v) (wakeup r));      (* 2. then wait for (f v), copy into r *)
  r

(* for contrast — map (not in the file): g : 'a -> 'b, a VALUE *)
let map g p =
  let r = { state = Waiting [] } in
  on_resolve p (fun v -> wakeup r (g v));    (* ONE wait: g v is ready *)
  r
  • Two promises to await in turn: first p, then the promise that f v itself returns. One on_resolve each.
  • The second one is the monadic flatten: it forwards f v's eventual value into the placeholder r.
  • map needs only one — g v is a plain value. Functor (1 level) vs monad (2 levels).
OCaml 5 effects — the mechanism

Enter effects

type _ Effect.t += Await : 'a t -> 'a Effect.t | Yield : unit Effect.t

let await p = match p.state with
  | Done v -> v
  | Waiting _ -> Effect.perform (Await p)     (* suspend the fiber *)
  • perform suspends the current fiber and hands its continuation k to the enclosing handler.
  • The handler parks k as a waiter and re-enqueues it once the awaited promise resolves; resuming k re-enters under the same handler.
OCaml 5 effects — running fibers

The effect run

let run main =                       (* run : (unit -> 'a t) -> 'a *)
  let result = ref None in
  let fiber () =
    match result := Some (await (main ())) with
    | () -> ()                        (* main resolved *)
    | effect Await p, k ->            (* park k; resume when p resolves *)
        on_resolve p (fun v -> Queue.add (fun () -> Effect.Deep.continue k v) ready)
    | effect Yield, k -> Queue.add (fun () -> Effect.Deep.continue k ()) ready
  in
  Queue.add fiber ready;
  loop ();                           (* the SAME ready/timers loop as V1 *)
  match !result with Some v -> v | None -> failwith "run: deadlock"
  • Same loop as V1; the only additions are the handler and a fiber wrapper around main.
  • It takes unit -> 'a t, not 'a t: the awaits must happen inside the fiber.
Version 2 — effects, broken

It breaks the semantics minilwt_eff_break.ml

let bind p f = f (await p)
  • Tempting: the shortest possible bind, and the native stack survives the await.
  • But on a pending promise await suspends the whole fiber: bind does not return until p resolves.
  • both (a >>= f) (b >>= g) cannot start b until a is done — the branches run in series.
The bug, on the clock

Same program, two timings

let task label d =
  let* () = sleep d in Printf.printf "%s done\n" label; return ()
let both a b = let* () = a in let* () = b in return ()

run (fun () -> both (task "a" 0.15) (task "b" 0.15))
V1 & V3 (preserved)
0.15 s — concurrent
V2 (broken)
0.30 s — serialised

Both task calls run (starting both timers) before both does — unless bind suspends on the first await. That single difference is the whole gap.

Version 3 — effects, preserved

Semantics preserved minilwt_eff_keep.ml — "mbind"

(* same effect machinery as V2, but bind is V1's, byte for byte *)
let bind p f =
  let r = { state = Waiting [] } in
  on_resolve p (fun v -> on_resolve (f v) (wakeup r));
  r
  • Non-blocking again → concurrency preserved: both is back to 0.15 s.
  • await is still there as an opt-in direct-style escape hatch (loops, native try/with, real backtraces) that bind never uses.

Takeaways

What the three teach

  • The preserving effect bind is the classic bind. Effects don't replace the monad — they add a direct-style await and a lean scheduler underneath.
  • The one-liner bind = f (await p) is seductive but silently changes what a program means.
  • The monadic type 'a t keeps marking async in the signature — the thing Lwt users rely on.
  • This is the boiled-down model of a larger experiment: a full effects-based Lwt core with an io_uring back end.
